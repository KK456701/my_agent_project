### 案例 2

- **来源 PR**: Demo: 用户登录模块

- **严重程度**: medium
- **描述**: _CONFIG_CACHE 作为全局字典缓存配置，存在与 _TOKEN_CACHE 相同的线程安全和可测试性问题。此外，它没有 TTL（生存时间）和淘汰策略，导致配置更新后缓存不会失效，可能返回过时的配置值。对于敏感配置（如 API Key），存储在全局内存中也没有访问控制。
- **建议**: 1. 使用带 TTL 的缓存库（如 cachetools.TTLCache）。2. 考虑使用 Redis 等外部缓存服务，支持过期和淘汰。3. 将配置管理抽象为 ConfigService 类，支持缓存刷新和监听配置变更。4. 敏感配置应使用专门的密钥管理服务（如 Vault）。
